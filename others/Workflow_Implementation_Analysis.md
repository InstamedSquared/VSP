# VSP Workflow & Structure Implementation Analysis

This analysis synthesizes both the `Project_Process_Overview.md` and `Structure.txt` to provide a comprehensive roadmap of how the VSP Workforce Management platform will be implemented within your current Node.js/React architecture.

## 1. System Architecture Foundation

Your current stack (React frontend, Node/Express backend, MySQL/MariaDB database) is well-suited for this multi-portal architecture. 

**Authentication & Roles (`UserModel.js`)**
The core of your application requires routing three distinct user types. Your existing `UserModel.js` is already designed to handle this by unioning three separate tables (`users` for Admin, `employees`, and `clients`). 
- Admin users (`t=0`) will be routed to the **Admin Panel**.
- Employees (`t=1`) will be routed to the **Employee Dashboard**.
- Clients (`t=2`) will be routed to the **Client Portal**.

## 2. Module Implementations

### Module 1: Client Portal
The React frontend will need a dedicated set of routes (e.g., `/client/*`) protected by role-based middleware.
- **Dashboard & Staff Management:** React components to list active assigned staff, pulling from an `employee_assignments` junction table in the backend. 
- **Billing Module:** Integrating Stripe Elements in React for the UI, while the Node.js backend handles webhooks and auto-generation of PDF invoices (using a library like `pdfkit` or `puppeteer`).
- **Contract & Training Repository:** Using your server's storage (`private_storage`) to store uploaded SOPs, NDAs, and BAAs. The React frontend can leverage Tiptap (`@tiptap/react`) for rich text training instructions and embedded YouTube/Loom videos.

### Module 2: Employee Dashboard
Protected React routes (e.g., `/employee/*`).
- **Profile & Payslips:** A secure area where employees upload resumes and IDs. Payslips will be generated by the admin/system and served as downloadable PDFs through a secure Express route.
- **LMS / Courses:** Since you have Tiptap installed, creating course material with rich text and multimedia is straightforward. You will need a `courses` table and an `employee_course_progress` table to track quiz scores and completion.
- **Reprofile System:** A simple status toggle in the React UI that updates the employee's status to "Available for Reprofile" in the database, immediately making them visible to the Admin/Recruitment panel.

### Module 3: Admin Panel
Protected React routes (e.g., `/admin/*`).
- **Recruitment Dashboard:** A Kanban-style board in React (potentially using a drag-and-drop library like `react-beautiful-dnd` or the installed `react-draggable`) to move applicants through pipeline stages (Applied -> Screening -> Assessment -> Hired).
- **Compliance & Communications:** A broadcast messaging system. The backend will need a `communications` table, and the frontend will display alerts or mandatory acknowledgment popups. A cron job in Node.js can handle daily checks for NDA/HIPAA expirations.

### Module 4 & 5: Third-Party Integrations
- **Hubstaff:** A dedicated backend service (`HubstaffService.js`) that runs via node-cron, polling the Hubstaff API for hours, attendance, and screenshots. To prevent rate-limiting, this data should be cached locally in your database or the existing Redis instance.
- **Stripe:** Implemented via `StripeService.js`. Handles recurring billing, payment capture, and webhooks for failed payment notifications.

### Module 6: Hiring Portal (Public Facing)
- **Public Routes:** Unauthenticated routes (e.g., `/careers` or `/hire`) where prospective clients can submit staffing requests and applicants can upload resumes. These will feed into `client_requests` and `applicants` tables, which the Admin Panel will monitor.

### Modules 7 & 8: Floating Bench & Replacement Systems
- **Floating Bench Management:** An internal view in the Admin Panel filtering the `employees` table by `status = 'floating'`. The `QueryBuilder.js` can be used to execute complex skill-matching queries against an `employee_skills` table.
- **Replacement Requests:** A ticketing system in the Client Portal. When a client submits a replacement request, it triggers an email notification to HR (via NodeMailer) and opens a transition workflow in the Admin Panel.

## 3. Recommended Next Steps for Development

When you are ready to begin development, the recommended approach is:
1. **Database Schema:** Finalize and create the necessary tables (e.g., `invoices`, `leave_requests`, `applicants`, `courses`, `communications`).
2. **Backend Integrations:** Build the Hubstaff and Stripe service classes first, as they form the backbone of the billing and productivity modules.
3. **API & Routes:** Create the Express CRUD routes for the three portals.
4. **Frontend UI:** Build the React layouts for the Admin, Client, and Employee dashboards.
